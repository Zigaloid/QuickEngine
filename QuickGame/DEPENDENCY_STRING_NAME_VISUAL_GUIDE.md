## API Comparison: Template vs String

```
???????????????????????????????????????????????????????????????
? API Comparison                                              ?
?????????????????????????????????????????????????????????????
? Aspect                 ? Template       ? String API       ?
??????????????????????????????????????????????????????????????
? Usage                  ? Template<T>    ? String literal   ?
? Type Safety            ? Compile-time   ? Runtime          ?
? Error Detection        ? Compiler       ? Return value     ?
? Data-driven Setup      ? No             ? Yes              ?
? Configuration Files    ? Not practical  ? Yes              ?
? Plugin System         ? Limited        ? Excellent        ?
? Performance           ? No overhead    ? O(1) lookup      ?
? Readability           ? Template heavy ? Clear            ?
??????????????????????????????????????????????????????????????

Recommendation:
  - Core systems: Use Template API
  - Plugins/Config: Use String API
  - Mixed: Use both!
```

## Summary: What Happens When...

```
SCENARIO 1: Register and Add Dependency
????????????????????????????????????????
manager->RegisterComponentType<CPhysicsBodyComponent>("CPhysicsBodyComponent")
  ?? Creates pool for CPhysicsBodyComponent
  ?? m_nameToTypeIndex["CPhysicsBodyComponent"] = type_index(...)

scheduler->RegisterComponentType<CPhysicsBodyComponent>()
  ?? m_phases.push_back(phase for CPhysicsBodyComponent)

scheduler->AddDependencyByName("CInput", "CPhysicsBodyComponent")
  ?? Lookup "CInput" ? found
  ?? Lookup "CPhysicsBodyComponent" ? found (from step 1)
  ?? Add to dependency list
  ?? Return true


SCENARIO 2: Unknown Component Type
??????????????????????????????????
scheduler->AddDependencyByName("CUnknown", "CPhysicsBodyComponent")
  ?? Lookup "CUnknown" ? NOT found (not registered)
  ?? Return false immediately


SCENARIO 3: Mixed Template and String API
??????????????????????????????????????????
scheduler->AddDependency<CRender, CTransform>()
  ?? Works as before (template resolution)

scheduler->AddDependencyByName("CInput", "CPhysics")
  ?? Works via string lookup

Result: Both dependency types active in scheduler
```
